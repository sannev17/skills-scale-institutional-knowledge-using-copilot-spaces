# OctoAcme Role Engagement Checklist

Use this checklist as a quick reference to identify **which roles are involved** at each phase of the project lifecycle, and **what they own or approve**. It is designed as a lightweight RACI-lite guide—not a strict governance framework—to help teams avoid gaps and overlaps in accountability.

**Legend**
- **R** = Responsible (does the work)
- **A** = Accountable (signs off / decision-maker)
- **C** = Consulted (provides input before decisions/work)
- **I** = Informed (kept up-to-date after decisions)

---

## Phase 1 — Initiation

| Activity | Project Manager | Product Manager | Developers | UX Designer | SME | Scrum Master | Business Analyst |
|---|---|---|---|---|---|---|---|
| Define problem statement | A | R | C | C | C | I | C |
| Identify stakeholders & sponsors | R/A | C | I | I | C | I | C |
| Draft project charter / one-pager | R | C | I | I | C | I | C |
| Initial scope & success metrics | C | A/R | I | C | C | I | R |
| Risk & constraint identification | R | C | C | I | A/R | I | C |

**Checklist**
- [ ] Problem statement is documented and approved by Product Manager
- [ ] Stakeholder list is complete and roles are assigned
- [ ] Project charter drafted and signed off
- [ ] SME consulted on domain constraints or regulatory requirements
- [ ] Business Analyst engaged to begin requirements elicitation

---

## Phase 2 — Planning

| Activity | Project Manager | Product Manager | Developers | UX Designer | SME | Scrum Master | Business Analyst |
|---|---|---|---|---|---|---|---|
| Backlog creation & prioritization | C | A/R | C | C | C | C | R |
| User research & persona validation | I | C | I | A/R | C | I | C |
| Requirements documentation | C | A | C | C | A/R | I | R |
| Sprint/iteration planning | C | C | R | C | I | A/R | C |
| Resource & timeline planning | A/R | C | C | C | I | C | C |
| Risk register creation | A/R | C | C | I | C | C | C |

**Checklist**
- [ ] Backlog items have acceptance criteria (owned by Business Analyst / Product Manager)
- [ ] UX Designer has completed initial wireframes / design direction
- [ ] SME has reviewed domain-specific requirements
- [ ] Sprint capacity confirmed with Scrum Master (Agile teams)
- [ ] Risk register populated and reviewed

---

## Phase 3 — Execution & Tracking

| Activity | Project Manager | Product Manager | Developers | UX Designer | SME | Scrum Master | Business Analyst |
|---|---|---|---|---|---|---|---|
| Feature implementation | I | I | A/R | C | C | I | C |
| Design review & UI validation | I | C | C | A/R | I | I | I |
| Domain/requirements Q&A | C | C | C | I | A/R | I | R |
| Daily standups / blockers | C | I | R | R | I | A/R | I |
| Status reporting | A/R | I | C | I | I | C | I |
| Scope change management | A | A/R | C | C | C | C | R |

**Checklist**
- [ ] Daily standups are running (Scrum Master owns for Agile teams)
- [ ] Design handoffs to Developers are complete and annotated
- [ ] SME consulted when domain questions arise during development
- [ ] Status report sent to stakeholders on schedule
- [ ] Change requests logged and impact assessed before approval

---

## Phase 4 — Release & Deployment

| Activity | Project Manager | Product Manager | Developers | UX Designer | SME | Scrum Master | Business Analyst |
|---|---|---|---|---|---|---|---|
| Release readiness review | A/R | C | C | C | C | C | C |
| Final UAT / acceptance sign-off | C | A/R | C | C | R | I | R |
| Deployment execution | I | I | A/R | I | I | I | I |
| Stakeholder announcement | C | A/R | I | I | I | I | I |
| Documentation & training materials | C | C | C | C | R | I | R |

**Checklist**
- [ ] UAT completed and sign-off obtained (Product Manager + Business Analyst)
- [ ] SME validated domain-specific features before go-live
- [ ] UX Designer confirmed UI fidelity meets approved designs
- [ ] Release notes and user documentation prepared
- [ ] Deployment runbook reviewed and approved

---

## Phase 5 — Retrospective & Continuous Improvement

| Activity | Project Manager | Product Manager | Developers | UX Designer | SME | Scrum Master | Business Analyst |
|---|---|---|---|---|---|---|---|
| Retrospective facilitation | C | C | R | R | C | A/R | C |
| Action item ownership | R | C | C | C | C | C | C |
| Process documentation updates | A/R | I | C | I | I | C | R |
| Lessons-learned summary | R | C | C | C | C | C | C |

**Checklist**
- [ ] Retrospective held within one week of release
- [ ] Action items assigned with owners and due dates
- [ ] Process docs updated to reflect agreed improvements
- [ ] Lessons-learned summary shared with stakeholders
